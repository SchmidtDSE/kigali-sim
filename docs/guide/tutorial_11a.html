Tutorial 11a: Agentic AI (JVM)

Using AI agents to autonomously run and iterate on simulations through the JVM.

Motivation

In Tutorial 11, we saw how AI assistants can help write QubecTalk code. Some AI assistants can go further: when equipped with code-execution capabilities, they can act as agents — reading data files, running simulations, evaluating outputs, and proposing iterative refinements, all autonomously. This tutorial shows how to set up such a workflow using the Kigali Sim JVM command-line interface.

The core principle throughout this tutorial is: AI assistants can help with execution, but the decision-making remains in your hands. Agentic workflows can move fast — it is essential to pause and review what the agent has produced at every stage before asking it to continue.

Prerequisites

This tutorial requires the following before you begin:

  1. An AI assistant with code-execution capabilities. The agent must be able to run shell commands, read files, and write files on your machine. Claude (with the computer use or code-execution tool enabled) and Mistral (with code execution) are examples we have tested. Note that code-execution features may require a paid plan — check your AI provider's documentation.
  2. The Kigali Sim fat JAR. Download kigalisim-fat.jar and save it somewhere accessible on your machine. You can also find the download link at the bottom of kigalisim.org.
  3. Java 21 or higher. Install Java through Adoptium or another provider. You can verify your installation by running java -version in a terminal.
  4. Network access to kigalisim.org. The agent will need to read the Kigali Sim language reference. If your organisation restricts internet access, you may need to allowlist kigalisim.org or download llms-full.txt and provide it as an attachment.

A note on privacy: Any data files and simulation code you share with an AI assistant will be sent to that provider's servers. Please review your AI provider's privacy policy and terms of service, particularly if your data contains sensitive national statistics.

Project Setup

Agentic workflows perform best when the AI has clear, structured context about the modelling problem before it starts running simulations. We recommend creating a short project document — a plain text or Markdown file — that the agent can read at the start of each session.

Your project document should describe:

  • The country and sector being modelled (e.g. "ABC Country, Domestic Refrigeration").
  • The substances in scope and their roles (e.g. HFC-134a as the incumbent substance, R-600a as the low-GWP alternative).
  • The data sources available and what each contains (e.g. a Socioeconomics sheet with GDP and population projections, a Metadata sheet with GWP values, a Census sheet with equipment population estimates, an Import/Export sheet).
  • Key modelling constraints (year range, policy targets, any known data quality issues).
  • The path to the Kigali Sim JAR and where to write output CSV files.

Once your project document is ready, start the agent with a message like this:

Hello! Please read https://kigalisim.org/llms-full.txt?v=20260302 to learn about Kigali Sim and QubecTalk. Then read my project document [attach file]. I will give you tasks that involve running the Kigali Sim JAR at [path/to/kigalisim-fat.jar] using: java -jar [path/to/kigalisim-fat.jar] run -o output.csv script.qta. Please confirm you understand the setup before we begin.

Wait for the agent to confirm it has read both documents and understood the setup. Correct any misunderstandings before moving on — a small error in the agent's initial understanding of the data structure can propagate through all subsequent steps.

Step 1: Historic Data Visualisation

The first task is to ask the agent to build an initial simulation that reproduces historic consumption data. This serves two purposes: it validates that the agent has understood your data correctly, and it gives you a baseline before any forward projection is added. 

Please read the attached data file. Using the Kigali Sim JAR, build an initial simulation for HFC-134a and R-600a in Domestic Refrigeration that reproduces the historic consumption data from [start year] to [end year]. Run the simulation, write the results to historic_baseline.csv, and show me the output. Flag any data points that look unusual or that required assumptions on your part. Then pause and wait for my review.

When the agent responds, open the CSV and check whether the simulated values match your source data. Pay attention to units — confirm whether the output is in mt / year or kg / year as expected. Correct any discrepancies before proceeding.

Important: Do not ask the agent to continue until you are satisfied that the historic baseline is correct. Errors in the baseline will carry forward into the projection and policy steps.

Step 2: Forward Projection

Once the historic baseline looks correct, ask the agent to extend the simulation into the future. For HFC-134a, a GDP-linked growth model is appropriate. For R-600a, curve-fitting to the historic trend often works well for a business-as-usual projection. 

Please extend the simulation to [future year]. For HFC-134a, tie the growth rate to the GDP projections in the Socioeconomics sheet. For R-600a, fit a trend to the historic data. Run the extended simulation, write results to bau_projection.csv, and explain the assumptions you made for each substance. Pause and wait for my review before continuing.

Review the projection logic carefully. Ask yourself:

  • Does the HFC-134a growth rate feel plausible given GDP projections?
  • Does the R-600a trend line reflect a realistic technology uptake curve?
  • Are retirement and recharge rates consistent with what is known about the equipment population?

If anything looks wrong, ask the agent to explain its reasoning or to try a different approach. The agent should not move on to policy modeling until you have approved the business-as-usual projection.

Step 3: Policy Modeling

With an approved business-as-usual simulation in place, ask the agent to add policy scenarios. The example below adds two policies commonly found in Kigali Amendment Implementation Plans: a sales cap displacing the high-GWP substance, and a recycling programme. 

Please now add two policy scenarios on top of the Business as Usual:

1. A cap on HFC-134a sales of 80% starting in [policy year], displacing to R-600a.
2. A recycling programme starting in [policy year] capturing 20% of HFC-134a at the time of servicing, with a yield loss of 50%.

Please add each policy as a separate simulation so we can compare all three scenarios. Run all simulations, write results to policy_comparison.csv, and summarise the emissions delta between BAU and each policy scenario. Pause after running.

Check that the policy logic is correctly reflected in the QubecTalk code before accepting the results. In particular, verify that:

  • The cap is applied to the correct substance and application.
  • The displacement target substance is R-600a (not left unspecified).
  • The recycling programme captures at the correct stage (servicing, not end-of-life, in this example).

Reviewing and Validating Outputs

Before concluding the agentic session, ask the agent to summarise its own work and flag any remaining uncertainties: 

Please summarise the key assumptions you made throughout this modelling session and flag anything you are uncertain about or that you think I should double-check.

Use this as a starting point for your own review. At minimum, verify the following:

  • Substance totals and units are consistent throughout all output CSVs.
  • The GWP values used match those in your Metadata sheet.
  • The simulation year range is correct in all scenarios.
  • The policy deltas between scenarios make intuitive sense (e.g. the recycling scenario should show lower net consumption than BAU).
  • The QubecTalk code can be loaded into the Kigali Sim web editor without errors if you want to explore results visually.

The agent's summary does not replace your own review. You are responsible for the model outputs.

Conclusion

This tutorial demonstrated an agentic AI workflow for Kigali Sim using the JVM command-line interface. The agent handled the mechanical work of writing QubecTalk, running simulations, and reading CSV outputs — but the substance of the model (assumptions, parameters, policy logic) remained under your control at every stage.

Key takeaways:

  • Structured context matters: a well-written project document reduces errors and misunderstandings early.
  • Pause and review at every step: agentic workflows can move fast, and errors compound if not caught early.
  • The agent flags; you decide: treat the agent's uncertainty flags as prompts for your own scrutiny, not as a complete list of issues.

For browser-based AI assistance without code execution, see Tutorial 11. For more on the command-line interface used in this tutorial, see Tutorial 16.
